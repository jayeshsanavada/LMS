How to Read This Diagram

LayerColorRoleModules
API Layer Brown Entry point. Every external HTTP request enters through M15 and is routed to the appropriate business module. M15 API Layer
Core Blue Primary business value. These modules contain the features users directly interact with. M1 User Mgmt, M2 Training, M3 Assignment Engine, M4 Sessions
Support Orange Enable core flows. These modules run in the background or provide data views — they don't own the primary business objects. M5 Notifications, M6 Reporting, M7 Audit, M8 Integrations
Platform Green Reusable foundation. Cross-cutting capabilities used by both Core and Support modules. M10 Auth, M11 AuthZ, M12 Search, M13 File Mgmt, M14 Admin Config, M16 BG Jobs
Deferred Grey (dashed) Phase 2 only — not built or tested in Phase 1. M9 Migration

Key Architectural Rules

RuleWhat It Means
All async work runs through M16 Background JobsNotifications (M5), Audit writes (M7), Integration syncs (M8), and Report exports (M6) are NEVER done inline in an API response. They are always dispatched to the Worker via M16.
Compliance logic lives only in M3 Assignment EngineNo other module evaluates compliance or applies probation gates. Even reporting reads pre-computed compliance_status — it does not re-evaluate.
M6 Reporting is read-onlyReporting reads data from Core modules' tables. It has zero write operations — no inserts, no updates.
No cross-table writesEach module owns its DB tables. A module may read another module's tables but may never write to them directly.
M9 Migration is Phase 2Not built in Phase 1. Depends on M1, M2, M3, M4, M7 — all four core modules plus Audit.
