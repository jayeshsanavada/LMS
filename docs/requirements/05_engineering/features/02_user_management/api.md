# User Management — API Specification

---

## 1. Common Conventions

### Base Path
```
/api/v1
```

### Authentication
- All endpoints require a valid JWT Bearer token
- Token passed as: `Authorization: Bearer <token>`
- Roles extracted from JWT claim: `realm_access.roles`

### Correlation ID
- Every request must include `X-Correlation-ID` header (generated by middleware if absent)
- All responses echo back `X-Correlation-ID`

### Standard Response Envelope
```json
{
  "success": true,
  "data": {},
  "message": null,
  "errors": []
}
```

### Paginated List Response
```json
{
  "success": true,
  "data": {
    "items": [],
    "page": 1,
    "size": 20,
    "total": 120,
    "has_next": true
  },
  "message": null,
  "errors": []
}
```

### Pagination Parameters
```
?page=1&size=20
```

### Sorting Parameters
```
?sort_by=full_name&sort_order=asc
```

### Authorization Notation
- `ADMIN` / `HR` / `EMPLOYEE` — global roles from `realm_access.roles`
- Manager = any `EMPLOYEE` with direct reports in `user_hierarchy` (not a role)
- `SELF_ONLY` — access to own data only
- `TEAM_ONLY` — access limited to direct reports (depth = 1)
- `ORG_WIDE` — access across all users

### 2.1 List Users

**Endpoint**
```
GET /api/v1/users
```

**Purpose**
Returns a paginated list of LMS users. Used by Admin for user management and HR for org-wide visibility.

**Access**
- `ADMIN` — ORG_WIDE
- `HR` — ORG_WIDE (read only)

**Query Parameters**

| Parameter | Type | Description |
|---|---|---|
| `page` | integer | Page number (default: 1) |
| `size` | integer | Page size (default: 20, max: 100) |
| `status` | string | `ACTIVE` / `INACTIVE` / `EXITED` |
| `department` | string | Filter by department |
| `designation` | string | Filter by designation |
| `capability` | string | Filter by primary capability |
| `bu_head_id` | UUID | Filter by BU Head user ID |
| `manager_id` | UUID | Filter by reporting manager |
| `employment_phase` | string | Filter: `PROBATION` / `CONFIRMED` / `EXITED` |
| `include_inactive` | boolean | Include inactive users (default: false) |
| `sort_by` | string | `full_name` / `created_at` / `department` |
| `sort_order` | string | `asc` / `desc` |
| `q` | string | Name or email search (partial match) |

**Response: 200 OK**
```json
{
  "success": true,
  "data": {
    "items": [
      {
        "user_id": "uuid",
        "employee_id": "EMP-001",
        "full_name": "Jane Smith",
        "email": "jane@company.com",
        "department": "Engineering",
        "designation": "Senior Engineer",
        "capability": "Backend",
        "bu_head_id": "uuid",
        "bu_head_name": "Ravi Patel",
        "capabilities": ["Java Backend", "Spring Boot", "AWS"],
        "global_role": "EMPLOYEE",
        "employment_status": "ACTIVE",
        "employment_phase": "CONFIRMED",
        "is_active": true,
        "joining_date": "2023-01-15",
        "source_system": "ZOHO"
      }
    ],
    "page": 1,
    "size": 20,
    "total": 85,
    "has_next": true
  },
  "message": null,
  "errors": []
}
```

**Error Responses**

| HTTP | Error Code | Condition |
|---|---|---|
| 403 | `ACCESS_DENIED` | Non-admin/HR user attempts access |

### 2.2A Update Own Profile

**Endpoint**
```
PATCH /api/v1/users/me
```

**Purpose**
Allows any authenticated user to update their own editable profile fields. Identity fields (employee_id, email) and system-managed fields (department, designation, capability) are read-only — those are sourced from Zoho / Employee DB and can only be corrected by Admin.

**Access**
- Any authenticated user — SELF_ONLY

**Request Body** (all fields optional — send only fields to update)
```json
{
  "full_name": "Rahul Sharma",
  "phone": "+91 98765 43210",
  "location": "Bangalore"
}
```

**Editable fields:** `full_name`, `phone`, `location`
**Read-only (ignored if sent):** `employee_id`, `email`, `department`, `designation`, `capability`, `global_role`

**Response: 200 OK**
```json
{
  "success": true,
  "data": {
    "user_id": "uuid",
    "full_name": "Rahul Sharma",
    "phone": "+91 98765 43210",
    "location": "Bangalore",
    "updated_at": "2026-04-07T10:00:00Z"
  },
  "message": "Profile updated.",
  "errors": []
}
```

**Side Effects**
- Audit event `USER_SELF_PROFILE_UPDATED` emitted

### 2.4 Create User (Manual)

**Endpoint**
```
POST /api/v1/users
```

**Purpose**
Allows Admin to manually create a user. Available at any time — not restricted to exceptions. Manually created users are tagged `source_system = MANUAL` and overwritten by Zoho sync on matching `employee_id`.

**Access**
- `ADMIN` only

**Request Body**
```json
{
  "employee_id": "EMP-099",
  "email": "new.user@company.com",
  "full_name": "New User",
  "department": "Engineering",
  "designation": "Engineer",
  "capability": "Frontend",
  "bu_head_id": "uuid",
  "capabilities": ["Frontend", "React"],
  "location": "Ahmedabad",
  "joining_date": "2026-04-06"
}
```

**Validation Rules**
- `employee_id` must be unique
- `email` must be unique and valid format
- `full_name`, `department` are required
- `global_role` defaults to `EMPLOYEE` — not settable on creation

**Response: 201 Created**
```json
{
  "success": true,
  "data": {
    "user_id": "uuid",
    "employee_id": "EMP-099",
    "email": "new.user@company.com",
    "full_name": "New User",
    "global_role": "EMPLOYEE",
    "source_system": "MANUAL",
    "is_active": true,
    "created_at": "2026-04-06T10:00:00Z"
  },
  "message": "User created successfully.",
  "errors": []
}
```

**Error Responses**

| HTTP | Error Code | Condition |
|---|---|---|
| 400 | `VALIDATION_ERROR` | Missing required fields or invalid format |
| 409 | `DUPLICATE_EMPLOYEE_ID` | employee_id already exists |
| 409 | `DUPLICATE_EMAIL` | email already exists |
| 403 | `ACCESS_DENIED` | Non-admin user attempts creation |

**Side Effects**
- Audit event `USER_CREATED_MANUAL` emitted
- Default `role_mappings` row created with `EMPLOYEE`
- Keycloak user provisioning triggered (async)

### 2.5 Update User Attributes (Admin Override)

**Endpoint**
```
PATCH /api/v1/users/{user_id}
```

**Purpose**
Allows Admin to correct specific profile fields when source system data (Zoho/Employee DB) is incorrect. Overridden fields are tagged `manually_overridden = true` and not overwritten by sync until Admin explicitly clears the override.

**Access**
- `ADMIN` only

**Path Parameters**

| Parameter | Type | Description |
|---|---|---|
| `user_id` | UUID | Target user ID |

**Request Body** (all fields optional — only send fields to update)
```json
{
  "full_name": "Jane Smith-Updated",
  "department": "Product Engineering",
  "designation": "Lead Engineer",
  "location": "Bangalore",
  "bu_head_id": "uuid",
  "clear_overrides": false
}
```

**Notes**
- `clear_overrides: true` removes all manual override flags — next sync will overwrite these fields
- Overridable fields: `full_name`, `department`, `designation`, `location`, `bu_head_id` — identity fields (`employee_id`, `email`) are not overridable

**Response: 200 OK**
```json
{
  "success": true,
  "data": {
    "user_id": "uuid",
    "updated_fields": ["department", "designation"],
    "manually_overridden_fields": ["department", "designation"],
    "updated_at": "2026-04-06T10:00:00Z"
  },
  "message": "User attributes updated.",
  "errors": []
}
```

**Error Responses**

| HTTP | Error Code | Condition |
|---|---|---|
| 400 | `VALIDATION_ERROR` | Attempting to override non-overridable fields |
| 404 | `USER_NOT_FOUND` | User does not exist |
| 403 | `ACCESS_DENIED` | Non-admin user |

**Side Effects**
- Overridden fields tagged in `user_field_overrides` table
- Audit event `USER_ATTRIBUTE_OVERRIDDEN` emitted

### 2.6 Deactivate User

**Endpoint**
```
POST /api/v1/users/{user_id}/deactivate
```

**Purpose**
Deactivates a user in LMS. Cascades to pending approvals and future sessions.

**Access**
- `ADMIN` only

**Path Parameters**

| Parameter | Type | Description |
|---|---|---|
| `user_id` | UUID | Target user ID |

**Request Body**
```json
{
  "reason": "Employee exit - resignation"
}
```

**Response: 200 OK**
```json
{
  "success": true,
  "data": {
    "user_id": "uuid",
    "is_active": false,
    "deactivated_at": "2026-04-06T10:00:00Z"
  },
  "message": "User deactivated successfully.",
  "errors": []
}
```

**Error Responses**

| HTTP | Error Code | Condition |
|---|---|---|
| 404 | `USER_NOT_FOUND` | User ID does not exist |
| 409 | `USER_ALREADY_INACTIVE` | User is already deactivated |
| 403 | `ACCESS_DENIED` | Non-admin user attempts deactivation |

**Side Effects**
- `users.is_active` set to false
- Pending approval requests by/for user auto-rejected (`reason: USER_DEACTIVATED`)
- Future session participations removed
- Audit event `USER_DEACTIVATED` emitted

### 2.7 Reactivate User

**Endpoint**
```
POST /api/v1/users/{user_id}/reactivate
```

**Purpose**
Reactivates a deactivated user. Used for rehires or mistaken deactivations. Restores LMS access and re-enables Keycloak account. Triggers mandatory training re-evaluation.

**Access**
- `ADMIN` only

**Path Parameters**

| Parameter | Type | Description |
|---|---|---|
| `user_id` | UUID | Target user ID |

**Request Body**
```json
{
  "reason": "Employee rehired"
}
```

**Response: 200 OK**
```json
{
  "success": true,
  "data": {
    "user_id": "uuid",
    "is_active": true,
    "reactivated_at": "2026-04-06T10:00:00Z"
  },
  "message": "User reactivated successfully.",
  "errors": []
}
```

**Error Responses**

| HTTP | Error Code | Condition |
|---|---|---|
| 404 | `USER_NOT_FOUND` | User does not exist |
| 409 | `USER_ALREADY_ACTIVE` | User is already active |
| 403 | `ACCESS_DENIED` | Non-admin user |

**Side Effects**
- `users.is_active` set to true
- Keycloak account re-enabled (async)
- Mandatory training re-evaluation triggered (async)
- Previously auto-rejected approvals are NOT reinstated
- Audit event `USER_REACTIVATED` emitted

### 2.8 Bulk Deactivate Users

**Endpoint**
```
POST /api/v1/users/bulk-deactivate
```

**Purpose**
Deactivates multiple users in a single operation. Applies the same cascade logic as individual deactivation for each user.

**Access**
- `ADMIN` only

**Request Body**
```json
{
  "user_ids": ["uuid-1", "uuid-2", "uuid-3"],
  "reason": "Department offboarding - project closure"
}
```

**Validation Rules**
- Maximum 100 users per bulk operation
- All provided `user_ids` must exist

**Response: 200 OK**
```json
{
  "success": true,
  "data": {
    "deactivated_count": 3,
    "failed_count": 0,
    "failed_ids": []
  },
  "message": "3 users deactivated successfully.",
  "errors": []
}
```

**Error Responses**

| HTTP | Error Code | Condition |
|---|---|---|
| 400 | `EXCEEDS_BULK_LIMIT` | More than 100 user IDs provided |
| 403 | `ACCESS_DENIED` | Non-admin user |

**Side Effects**
- Same cascade effects as `POST /users/{user_id}/deactivate` applied per user
- Audit event `USER_DEACTIVATED` emitted once per user in the batch (N events for N users, not a single bulk event)

### 2.12 Trigger User Sync

**Endpoint**
```
POST /api/v1/users/sync
```

**Purpose**
Manually triggers a Zoho and/or Employee DB sync job. Job runs asynchronously. Returns the job ID for status tracking.

**Access**
- `ADMIN` only

**Request Body**
```json
{
  "include_zoho": true,
  "include_employee_db": true
}
```

**Response: 202 Accepted**
```json
{
  "success": true,
  "data": {
    "job_id": "uuid",
    "status": "PENDING",
    "triggered_at": "2026-04-06T10:00:00Z"
  },
  "message": "Sync job triggered. Track status via integration health dashboard.",
  "errors": []
}
```

**Error Responses**

| HTTP | Error Code | Condition |
|---|---|---|
| 409 | `SYNC_ALREADY_RUNNING` | A sync job for the same source is already in progress |
| 403 | `ACCESS_DENIED` | Non-admin user |

**Side Effects**
- Background job created in `background_jobs` table
- Audit event `USER_SYNC_COMPLETED` emitted on job completion (not at trigger time)

## 3. Audit Event Codes Reference

| Event Code | Trigger | Emitted From |
|---|---|---|
| `USER_CREATED_SYNC` | User provisioned via Zoho sync | Sync worker |
| `USER_CREATED_MANUAL` | Admin manually creates user | POST /users |
| `USER_DEACTIVATED` | User deactivated | POST /users/{id}/deactivate or sync |
| `USER_REACTIVATED` | User reactivated | POST /users/{id}/reactivate |
| `USER_ROLE_CHANGED` | Admin changes global role | Admin module |
| `USER_HIERARCHY_UPDATED` | Reporting manager changes via sync | Sync worker |
| `USER_ATTRIBUTE_OVERRIDDEN` | Admin overrides profile field | PATCH /users/{id} |
| `USER_SELF_PROFILE_UPDATED` | User updates own profile | PATCH /users/me |
| `USER_SYNC_COMPLETED` | Sync job finishes | Sync worker |
| `USER_SYNC_CONFLICT` | Conflicting field values across sources | Sync worker |
| `USER_CAPABILITIES_UPDATED` | Admin updates manual capabilities | PUT /users/{id}/capabilities |
| `USER_CAPABILITIES_SYNCED` | On-demand capability sync from Employee DB | POST /users/{id}/capabilities/sync |
